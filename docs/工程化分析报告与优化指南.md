## 一、定位：本仓在三仓体检中的位置

手册第 1 章对三仓做了体检。结论里 **newide-scaffold 是三仓中工程化最好的**：

手册第 9 章给本仓的"额外重点"只有三条，且明确说**只差临门一脚**：

> 把 `src/*/contract.ts` 抽成契约种子；标签全角冒号改半角；**补 CI（工程化最好，差临门一脚）**。

因此本次改造目标不是推倒重来，而是**补齐基线缺口**，让本仓达到手册第 9 章定义的"协同开发工程基线"。

**已达标项（保留，不动）**：`pnpm` 包管理、ESLint flat config、Prettier、`tsconfig` `strict`+`noUncheckedIndexedAccess`+`exactOptionalPropertyTypes`、模块边界（README 已写清）、契约写成 `contract.ts`。

### 说明 A：Prettier 配置对齐取舍

手册第 4.4 基线为 `singleQuote: true, trailingComma: 'all'`，本仓现状为 `singleQuote`(默认 false), `trailingComma: 'none'`。手册第 3 章强调"四仓格式一致"的价值在于 **跨仓统一**，故本次**对齐手册基线**（改为 `singleQuote: true` / `trailingComma: 'all'`）。这会触发一次全量 reformat（属一次性成本，由 lint-staged 持续保证）。

### 说明 B：tsconfig 取舍

手册建议 `module: "NodeNext"`，但本仓 README 明确是 **bundler 风格的单包 TS 项目**，现用 `module: ESNext` + `moduleResolution: Bundler`，与 `tsx` 运行链匹配。强切 `NodeNext` 会要求所有相对 import 带 `.js` 后缀，破坏现有代码。**决策：保留现有 module 设置**，只增量补 `noImplicitOverride: true`（纯增强、零破坏），其余 strict 项本仓已有。这是手册"按需裁剪 / 不照抄"原则的应用。

### 说明 C：仓内无法做的项（属流程/治理，不在本仓改造范围）

手册第 8 章的 **GitHub Org transfer、main 分支保护 Ruleset、required status check、Squash-only 合并策略**，以及第 6 章 **ade-contracts 契约仓**、第 7 章 **跨仓 Project 看板 / 里程碑 / 标签全角冒号修正**，均为 GitHub 平台侧 / 跨仓操作，需仓库 owner 在 GitHub Settings 操作，不在单仓文件改造内。本报告在第四节给出对应 checklist 供 owner 执行。

## 三、优化操作指南（具体、可复制）

按手册第 9 章建议顺序，先 1–6（脚手架与门禁），再 7–8（CI 与治理）。本仓 git 身份已配好（`shixianjie <shixianjie@stu.pku.edu.cn>`），症结 #6 不适用。

### 步骤 1 · 锁工具链

- 新增 `.nvmrc` → 内容 `20`
- `package.json` 增 `"packageManager": "pnpm@11.5.1"`（=本机已装版本，满足手册 `>=9`；锁定后 CI 的 `pnpm/action-setup` 会据此自动选版，无需在 workflow 里硬编码版本）、`"engines": { "node": ">=20", "pnpm": ">=9" }`
- `scripts` 补齐 `build` / `test` / `format:fix` / `verify`（`verify = lint && typecheck && test`）

### 步骤 2 · 升级 .gitignore（手册 4.2 完整版）

覆盖 deps/build、env & secrets（`.env*`，保留 `.env.example`）、本地状态/会话/归档（`*.session.json`/`*.zip`/`*.log`）、editor & OS。

### 步骤 3 · 新增 .editorconfig（手册 4.3）

`root=true` + utf-8 / lf / 末尾换行 / 去尾空格 / 2 空格缩进。

### 步骤 4 · 统一 ESLint + Prettier（手册 4.4）

- 装 `eslint-config-prettier`，在 flat config 末尾追加 `prettier` 以关闭冲突规则，并显式 `ignores: ['dist/','node_modules/','coverage/']`。
- `prettier.config.js` 对齐 `singleQuote: true` / `trailingComma: 'all'`（printWidth 100、semi 保持）。

### 步骤 5 · tsconfig 增强（手册 4.5，按说明 B 裁剪）

增 `noImplicitOverride: true`（其余 strict 项已具备；module 保持 bundler 风格）。

### 步骤 6 · 预提交门禁（手册 4.6）

```bash
pnpm add -D husky lint-staged @commitlint/cli @commitlint/config-conventional eslint-config-prettier vitest
pnpm exec husky init
```

- `.husky/pre-commit` → `pnpm exec lint-staged`
- `.husky/commit-msg` → `pnpm exec commitlint --edit "$1"`
- `package.json` 增 `lint-staged`：`*.{ts,tsx,js,json,md}` → `prettier --write`；`*.{ts,tsx}` → `eslint --fix`
- `commitlint.config.js` → `extends: ['@commitlint/config-conventional']`
- 增一个冒烟测试（如 `test/smoke.test.ts`）让 `pnpm test`(vitest) 有内容可跑，`pnpm verify` 才能绿。

### 步骤 7 · CI（手册 4.7）★

`.github/workflows/ci.yml`：`pull_request` + `push:[main]` 触发，checkout → pnpm/action-setup → setup-node(20, cache pnpm) → `pnpm install --frozen-lockfile` → `pnpm verify`。

### 步骤 8 · 治理文件（手册 4.8 + 第 2 章）

`.github/PULL_REQUEST_TEMPLATE.md`、`.github/ISSUE_TEMPLATE/task.md`、`.github/ISSUE_TEMPLATE/bug.md`、`.github/CODEOWNERS`、`CONTRIBUTING.md`、`LICENSE`。

## 四、需仓库 Owner 在 GitHub 平台手动执行（仓内无法完成）

来自手册第 6 / 7 / 8 章，建议本周内完成：

- [ ] **进 Org**：将仓库 transfer 到统一 Org（如 `ade-team`），旧地址会自动重定向，issue/PR/star/历史保留。
- [ ] **main 分支保护 Ruleset**：禁直推 → 必经 PR；合并前 ≥1 approve；CI `verify` 设为 required status check；要求分支与 main 最新；合并后自动删分支。
- [ ] **合并策略**：默认 Squash，关掉 merge commit，保持历史线性。
- [ ] **CODEOWNERS 占位替换**：把模板里的 `@module-owner` 等替换成真实 GitHub 账号。
- [ ] **契约单一可信源**：在 Org 下建 `ade-contracts` 仓，把本仓 `src/*/contract.ts` 抽成 11 份契约 markdown 种子 + 必埋点清单 `telemetry-required.md`。
- [ ] **看板/里程碑/标签**：接入跨仓 Project（Backlog→Ready→In Progress→In Review→Done）；标签统一为半角小写连字符（修正历史上的全角冒号 `area：c`）。
